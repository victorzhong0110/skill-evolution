# CEO Plan: Meta-Skill Evolution via External Evaluation

## Vision

### 10x Check

The 10x version doesn't just prevent meta-skill regression -- it actively discovers better meta-skill configurations through automated test case generation from real evolution runs, A/B testing of meta-skill variants in parallel, cross-meta-skill interaction analysis, and a real-time health dashboard showing performance trends over time. The user sees a system that gets smarter every time it runs, with evidence.

## Accepted Scope (added to this plan)

- `skill-evolution doctor` command: validates meta-skill file parsing, {k} placeholder presence, test suite completeness
- Rich progress output: real-time display of which meta-skill is being tested, current scores, candidate score deltas
- Changelog auto-generation: structured record per evolution cycle (what changed, why, which scores improved)
- TaskEvaluator plugin interface: users can implement custom evaluators (pytest runner, API checker, human judgment)
- Score trend visualization: export evolution history as CSV or ASCII chart showing score progression over time
- `--dry-run` flag for meta-evolve: run full pipeline without writing changes to meta-skill files

## Abbreviations

- **CC:** Claude Code estimated implementation time (with AI-assisted coding)
- **DeltaSignal:** A structured signal `{"type": "missing_knowledge"|"wrong_approach"|..., "description": str, "confidence": float}` produced by the Comparator to guide the Patcher. The Comparator compares baseline vs. candidate skill outputs and emits DeltaSignals describing why the candidate diverged.

## Core Plan (from /plan-eng-review)

### Phase 0: Bug Fixes

- **T1 (P1, CC: ~5min):** Add {k} placeholder to `strategy_generation.md`
  - Done when: `{k}` appears in the markdown file and `explorer.py:83` substitution produces a valid integer in the LLM prompt
- **T2 (P1, CC: ~20min):** Replace self-assessment with TaskEvaluator Protocol
  - Done when: executor.py uses TaskEvaluator instead of self-reporting, and at least one built-in evaluator exists
- **T2-ext (P1, CC: ~10min):** Extend TaskEvaluator into plugin interface (from D7)
  - Plugin interface contract: `evaluate(task_input, task_output) -> EvalResult` (sync). Async evaluators wrap via `asyncio.run()`. Human-in-the-loop evaluators block until stdin input.
  - Plugin discovery: evaluators are registered by subclassing `TaskEvaluator` and listing the module path in `EvolutionConfig.evaluator_class` (config-driven, not magic). Example evaluators provided: `GroundTruthEvaluator` (built-in), `PytestEvaluator` (example).
  - Done when: a user can configure a custom evaluator class in config and the pipeline uses it
- **T3 (P1, CC: ~15min):** Audit failure rollback + DeltaSignal feedback
  - T3 converts audit findings into DeltaSignals (see Abbreviations) so failed audits inform the next evolution round.
  - Done when: audit failure triggers automatic rollback to pre-patch state AND audit findings are converted to DeltaSignals passed to the next round

## Failure Modes

| Scenario | Impact | Mitigation |
|----------|--------|------------|
| LLM provider down during evolution | Evolution cycle fails mid-way | Existing snapshot allows manual rollback; evolution raises with clear error message naming the failed step |
| Test suite scoring function hangs | meta-test or meta-evolve blocks indefinitely | Add configurable timeout per scoring run (default: 120s, configurable per evaluator via `timeout_s` field). On timeout, treat as FAIL with reason "scoring timeout". LLM-backed evaluators should set higher timeouts. |
| Concurrent evolution on same target | Snapshot/rollback state corruption | Reject concurrent runs: acquire a file lock (`~/.skill-evolution/locks/{meta_skill_name}.lock`) before starting. If locked, raise "evolution already in progress for {name}" |
| Plugin evaluator crashes | executor.py propagates unhandled exception | Wrap evaluator calls in try/except; on crash, treat result as FAIL with the exception message |
| LLM returns empty/malformed response | Parser fails, potentially silent | Already addressed: auditor defaults to FAIL on unparseable output (TODOS.md P1). Extend same pattern to all LLM consumers |
| Partial evolution run (e.g., 3 of 5 strategies executed before crash) | Inconsistent state | Snapshot taken before evolution starts; any crash triggers rollback to snapshot |

## Complexity Assessment

The plan touches 8+ existing files and introduces 4+ new classes. Each new class earns its place: EvalCase/EvalResult are data models under 30 lines each, RegressionGate is a pure function under 40 lines, MetaSkillEvolver is the core orchestrator that makes the design work. The cherry-picked additions are incremental on top of these core classes. The complexity threshold comes from the /plan-ceo-review skill convention (challenges plans that touch >8 files or introduce >2 new classes).
